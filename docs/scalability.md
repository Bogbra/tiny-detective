# Scalability Notes

## Verification Scope

Everything below was verified against the **Firestore emulator** (Docker, `gcr.io/google.com/cloudsdktool/cloud-sdk:emulators`), not a real Firestore project — no real GCP project was available in this environment. This matters for two specific reasons, not just as a general caveat:

- **The emulator does not enforce composite indexes.** A query that needs a composite index runs fine locally and can fail against real Firestore with a `FAILED_PRECONDITION` until the matching index is created. None of the current queries (all single-field, e.g. "cases where status == approved") need one, but that's a property of today's query shapes, not something the emulator actually checked.
- **The emulator does not enforce Security Rules.** Locally, every read/write the backend issues succeeds regardless of what `firestore.rules` says (there is no deployed rules file yet — the backend's own `ADMIN_API_TOKEN` check is the only access control that actually runs today). Rules-based access control would need its own verification against a real project or the emulator's rules-testing mode, neither of which has happened.

Correct summary at the time this was written (Phase 7): **locally verified, index/rules behavior unverified.**

**Update (Phase 8, real deployment):** partially closed, precisely — not entirely. Today's query shapes were exercised against real (non-emulator) Firestore during the live deploy and worked with no `FAILED_PRECONDITION`, so the composite-index question is now actually answered for the queries that exist today, not just reasoned about. `firestore.rules` is now deployed (see [ADR-0006](architecture-decisions/ADR-0006-deployment-topology.md)) — but its *enforcement* was never actually exercised, because the backend's server SDK bypasses Security Rules entirely via IAM (that's the whole reason the rules file is a blanket deny). Nobody has attempted a direct client read/write against real Firestore to confirm the rules actually block it. So: index behavior — verified. Rules enforcement — deployed, but still structurally unverified, for a different reason than before (not "never deployed" anymore, but "never actually triggered by the one access pattern they exist to block").

## Data Model Choices Made for Read Efficiency

- **`daily_cases/{yyyy-mm-dd}` is a separate, date-keyed collection** rather than a query like "cases where publishedDate == today" against the larger `cases` collection. `GET /cases/daily` is a single document read by a known key, not a query — cheapest possible read shape, and immune to the composite-index question above since it's not a query at all.
- **Public case responses never include `isCulprit`/`privateReasoning`/the solution** (enforced at the API-response-mapping layer, not by separate documents) — keeps the hot path (a player loading a case) to one document read rather than joining across a "public case" and "private solution" collection.
- **Hint requests and attempts are separate collections keyed by their own IDs** (`hint_requests/{hintRequestId}`, `case_attempts/{attemptId}`), not subcollections nested under `players/{playerId}` — avoids Firestore's subcollection fan-out cost when e.g. an admin view needs "all attempts for a case" rather than "all attempts by a player."

## Known Limitations (Deferred, Not Solved)

- **Repository singletons are lazily constructed and cached per-process (`@lru_cache` in `app/api/dependencies.py`)** — one Firestore client per running backend process, not per-request. This is standard for a stateless Cloud Run-style deployment (the client is safe to reuse across requests, and Cloud Run itself handles horizontal scaling by running more instances, each with its own cached client) but hasn't been load-tested. See [ADR-0005](architecture-decisions/ADR-0005-firestore-data-model.md)'s addendum for why this used to be an eager module-level singleton and was changed.
- **No caching layer in front of Firestore** (e.g. Redis for `GET /cases/daily`, which is read far more often than it changes). Not needed at MVP traffic; would be the first thing to add if this were seeing real load.
- ~~No rate-limiting on hint requests beyond the per-case hint-count limit~~ — closed in Phase 8: `app/api/rate_limiting.py` adds a per-IP limit (5/minute) on `/hint` specifically, verified against the live deployment (6th rapid request from one caller gets `429`). See [ADR-0006](architecture-decisions/ADR-0006-deployment-topology.md).
- **AI cost control is still scope-limited (one OpenAI call per hint request, capped by the domain hint limit), plus now bounded by three deploy-time controls** (Phase 8): the rate limit above, `--max-instances=3` on Cloud Run, and a €10 GCP budget alert. None of these is a hard spend cap — the budget alert notifies, it doesn't stop billing — so "bounded" is accurate, "capped" would overclaim. ~~Case generation remains an offline/admin-triggered batch process, not player-triggered~~ — no longer true: `POST /cases/generate` (live case generation, see [ADR-0007](architecture-decisions/ADR-0007-live-case-generation.md)) is player-triggered and makes multiple real LLM calls per request. Its own cost exposure is bounded separately (a global daily success cap, a separate global daily attempt cap that specifically closes the "rejected attempts aren't otherwise bounded" gap, and a per-IP rate limit) rather than folded into this line, since the two features' AI-cost profiles are different enough to reason about independently.
- **The live case-generation feature's atomic daily counter is emulator-verified, not production-load-verified** — same honest scope as the composite-index/rules caveat above. A real concurrency bug (Firestore's default transaction retry budget getting exhausted under genuine parallel load) was found by actually hammering the emulator with 20 concurrent requests, not assumed away; fixed, but real GCP Firestore's behavior under production-scale concurrent traffic remains unexercised. See ADR-0007's decision 8.
- ~~Live case generation's real first-attempt pass rate (~5-10%) is well below what the feature was originally tuned for (~29%)~~ — closed via a redesign, not a bigger retry budget: the generator no longer invents its own deduction. A deterministic Python constraint solver decides who's guilty and verifies solvability before any LLM call; the LLM only writes prose around that fixed structure. Real measured pass rate after the redesign: **32.5%** (13/40 across two real 20-call batches), ~6.5x the old rate, at unchanged per-attempt cost. `MAX_ATTEMPTS_PER_REQUEST` was lowered from 10 to 5 accordingly. Every deterministically-built case has exactly 3 suspects and 4 clues by construction, so `assign_difficulty`'s heuristic now returns roughly the same difficulty for every live-generated case — a known, accepted trade-off, not fixed here. See ADR-0007's redesign addendum.
- **No BigQuery/analytics pipeline** — not needed until there's usage data worth aggregating beyond what Firestore itself can answer directly.
