# ADR-0007: Live Case Generation (Demo Feature)

## Status

Accepted

## Context

The game only ever showed the one curated daily case — no sense of a repeatable loop. The user asked for a "Generate a new case" button that runs the real AI pipeline (`tools/ai-content`'s generator + logic-judge + safety-judge) live, on a player's click, streams real pipeline progress via SSE, and — bounded by a hard global daily budget — saves the result into real Firestore as an immediately-playable case.

This is a deliberate, narrow, documented exception to the project's own "no unvetted AI case reaches players" rule, not a quiet workaround of it: opt-in (a player has to click a clearly-labeled button), both real AI judges still gate every candidate exactly as they do offline, capped at 50 successes/day, and every result is stored with `source="live_generated"` so it stays distinguishable and auditable. The actual daily case (`GET /cases/daily`) is never touched by this feature — it remains admin-curated only.

## Decisions

**1. Reuse `tools/ai-content` via a `uv` editable path dependency, not a reimplementation.** `services/api/pyproject.toml` now depends on `tools/ai-content` (`uv add --editable ../../tools/ai-content`). The generator, schema parser, rule validator, difficulty heuristic, and both AI judges are called directly from `app/infrastructure/ai/live_case_generator.py` — no duplicated logic. This required one upstream fix: `tools/ai-content/pyproject.toml` had no `[build-system]`/packages config, so setuptools' flat-layout auto-discovery choked on `evals/`/`prompts/` sitting next to the real `ai_content` package; added explicit `[tool.setuptools.packages.find] include = ["ai_content*"]`. The `"pytest" in sys.modules` guard in `ai_content.openai_client` carries over automatically since it's the same module object, not a copy — no new guard needed for that path. This does deepen the coupling between the two projects beyond ADR-0003's original "separate `uv` projects" framing, but duplicating hundreds of lines of real generation/validation logic (as opposed to the ~10-line OpenAI client guard ADR-0003 chose to duplicate) would have created two implementations of case-quality gates that could silently drift apart — a worse outcome than the coupling.

**2. Two independent daily counters, not one — a success cap the user asked for, plus an attempt cap they asked to be reconsidered.** `successCount` (default 50/day) is the user-visible product budget: only a candidate that passes both AI judges (plus the domain consistency check, see decision 4) counts against it, so a player is never penalized for a judge rejection outside their control. But counting only successes leaves total spend structurally unbounded — a rejected candidate still costs real tokens, and nothing else would cap how many rejected attempts happen in a day. `attemptCount` (default 300/day) closes that gap: every attempt, regardless of outcome, counts against it, and it's never exposed to the player (both failure modes show the same "daily generation quota reached" message). Both counters live in one Firestore document per UTC day (`rate_limits/case_generation_{yyyy-mm-dd}`), each gated by its own atomic transaction.

**3. Global cap and per-IP cap answer different questions — both are needed.** The global counters bound aggregate cost regardless of who's asking. A per-IP limit (`3/minute` on `POST /cases/generate`, reusing the existing shared `slowapi` `limiter` instance from `app/api/rate_limiting.py` with a second, independent `.limit()` string) stops one caller from burning the whole day's budget alone in minutes. Neither alone is sufficient — global-only lets one determined user exhaust the day for everyone; IP-only doesn't bound aggregate spend across many distinct callers.

**4. A deterministic domain check rides along inside the "logic check" step for free.** After the AI logic judge passes, the existing `CaseConsistencyPolicy` (Phase 2, pure, no I/O) runs against the constructed `DetectiveCase` before it's accepted — catching e.g. a candidate the AI judge missed where two suspects are both flagged as the culprit. This is not a fourth visible SSE step (the user asked for exactly three: generating/logic/safety) — a rejection here surfaces as a `logic_check: rejected` event, same as an AI judge rejection, and triggers the same retry loop.

**5. Hand-rolled SSE via FastAPI's built-in `StreamingResponse`, not a new `sse-starlette` dependency.** The event vocabulary is small (four step names, four statuses) — a 5-line async generator formatting `data: {...}\n\n` lines covers it without adding a dependency for something the standard library response class already supports. `POST`, not `GET`, despite "SSE" commonly implying the browser's native `EventSource` API (GET-only): the Flutter web client never uses `EventSource` at all — it reads the chunked `text/event-stream` body directly via `package:http`'s `Client.send()`, which works over any HTTP method.

**6. Blocking AI calls run via `asyncio.to_thread` inside the async generator.** `tools/ai-content`'s OpenAI calls are synchronous (the `openai` SDK's sync client). Without offloading them to a thread, one in-progress generation (15-30+ seconds worst case) would stall the event loop for every other concurrent request on the same Cloud Run instance — `/health`, `/hint`, everything.

**7. Both caps are environment-variable-configurable** (`CASE_GENERATION_DAILY_SUCCESS_CAP`, `CASE_GENERATION_DAILY_ATTEMPT_CAP`), specifically so the live-verification step could temporarily redeploy with a tiny cap to actually observe a real `429`, then restore the real defaults — not just asserted in tests.

**8. The atomicity claim is honestly scoped to what was actually tested — and a real gap was found by testing it.** "Atomic" is verified against the Firestore emulator's real transaction/conflict-retry behavior (`tests/integration/test_firestore_generation_quota_repository.py`), not against real GCP Firestore under genuine concurrent production load — no such load test is planned, matching the same honest "locally verified, X unverified" framing `docs/scalability.md` already uses for composite indexes. Testing this for real (20 concurrent threads hammering one document via `ThreadPoolExecutor`) found a genuine bug, not a hypothetical one: Firestore's Python client's default transaction retry budget (5 attempts) can be exhausted under real contention, raising `ValueError` instead of returning cleanly. Fixed by raising `max_attempts` to 15 *and* catching the exhausted-retries case explicitly, failing closed (deny the reservation) rather than crashing the request or risking a double-count. Under sufficiently extreme contention a request can still see "quota exhausted" when a slot was technically free — an accepted, documented trade-off for a low-traffic demo feature, not silently assumed away.

## Retry Count — Empirically Tuned, Not Guessed

`MAX_ATTEMPTS_PER_REQUEST = 5` was not an upfront estimate. Manual testing against the real API during implementation (`services/api`, real `OPENAI_API_KEY`, no fakes) ran the generator+judges pipeline 7 times: 2 passed both judges on the first attempt, 5 were rejected — every single rejection at the logic-consistency stage, zero at safety. That's a first-attempt success rate around 29% on a small sample (explicitly not claimed as a precise measurement — n=7 is an anecdote with a percent sign, not a statistic, the same caution this project applied to golden-set sizing in Phase 5). It was enough, though, to reject the original guess of 3 attempts (leaves a meaningful chance of full exhaustion) in favor of 5 (better eventual-success odds, still a bounded worst-case wait). It also directly validated why the attempt-cap (decision 2) matters: rejections are common enough in practice that an unbounded attempt count would be a real, not theoretical, cost exposure.

Real measured cost from that same testing: roughly **$0.00055–0.00064 per full attempt** (generator + logic judge + safety judge, gpt-4o-mini, current list pricing — same `$0.15`/`$0.60` per-1M-token constants `evaluate_cases.py` already uses). Worst-case daily spend at the attempt cap: `300 × ~$0.0006 ≈ $0.18/day` — comfortably under the existing €10 Cloud Run budget alert from ADR-0006, with wide margin for the pricing estimate to be off and still stay safe. `--max-instances=3` (also from ADR-0006) applies uniformly to this route too, no route-specific Cloud Run config needed.

**Confirmed against the actually-deployed production service, not just local runs** (the app's own `logger.info(...)` call in `generate_case.py`, read back via `gcloud logging read` — see the "found by testing" note below for why that logging almost didn't work at all): a real single-attempt success cost `$0.00053` (1947 prompt + 389 completion tokens); a real fully-exhausted 5-attempt request (all five rejected at the logic-consistency stage) cost `$0.00224` total (5570 prompt + 2336 completion tokens) — both in line with the local estimate above, if anything slightly cheaper since exhausted attempts here never reached the pricier safety-check stage. Real Firestore state after this session's live verification: `rate_limits/case_generation_2026-07-18` → `{successCount: 2, attemptCount: 8}`, confirming the two-counter document is actually being written correctly in production, not just in the emulator.

**The per-request logging almost didn't work at all — found by checking, not assumed.** The first live verification attempt produced no log output whatsoever for a completed generation. Cause: `logging.getLogger(__name__).info(...)` in `generate_case.py` was silently dropped — the root logger has no handler and defaults to `WARNING`, and uvicorn configures its own loggers (`uvicorn.error`/`uvicorn.access`) but not arbitrary application loggers. Fixed with one `logging.basicConfig(level=logging.INFO)` call in `app/main.py`; Cloud Run captures stdout/stderr automatically, so nothing more elaborate was needed. Redeployed, re-verified, logs appeared correctly.

**A deliberately-weakened prompt to force a rejection turned out not to be necessary.** The plan called for provoking one on purpose, expecting the tuned production prompt to rarely reject. In practice, live verification's very first real request produced a genuine rejection-then-success sequence unprompted, and a second request failed all 5 attempts outright (all five rejected at the logic-consistency stage) — both observed directly via the real SSE stream against the deployed URL, both showing the visible restart / clean "failed" terminal state exactly as designed. Given real rejections were this readily observable with the actual production prompt, deliberately weakening it for the same observation would have been redundant — noted here rather than silently skipped, since the plan explicitly called for it.

**The temporarily-lowered-cap 429 test was performed for real.** `gcloud run services update --update-env-vars=CASE_GENERATION_DAILY_SUCCESS_CAP=1` (below the day's already-real `successCount=2`) triggered an immediate, real `429 {"detail": "daily generation quota reached"}` from the live pre-check — confirmed via a direct `curl` against the production URL, not inferred. Reverted with `--remove-env-vars` immediately after; confirmed the service's env vars no longer show the override and `/health` still returns `200`.

## Consequences

- `DetectiveCase` gained a `source: str = "curated"` field (default preserves every existing call site — seed data, mappers, tests — unchanged); live-generated cases get `source="live_generated"` and `status=PublishStatus.LIVE` directly, skipping the normal draft→approve detour, since the two AI judges plus the domain policy are the approval for this path.
- `app/application/ports.py` gained `CaseGenerationAdapter` and `DailyGenerationQuotaRepository` — no existing repository pattern to mirror for the latter (confirmed via grep before building it), built fresh.
- The public `CaseResponse` contract now includes `source` — a small, deliberate transparency addition, not requested explicitly but consistent with the project's own "AI is visibly part of the product" framing.
- Regression risk: none to the existing daily-case flow — this feature adds a new route and a new use case, touches no existing use case's behavior. Verified: full backend suite green before and after, with and without the Firestore emulator.
- New regression surface worth naming: a future change to `tools/ai-content`'s generator/evaluator signatures would now break `services/api` too, since it's a real dependency, not a copy. This is the direct cost of decision 1's trade-off.
- **The Docker build context had to change, found by actually running the build, not by inspection.** `services/api/Dockerfile` used to build from `services/api` alone; with `tools/ai-content` as a path dependency (`../../tools/ai-content`, relative to `services/api/pyproject.toml`), that directory has to exist in the build context too, or `uv sync` fails with `cannot normalize a relative path beyond the base directory`. Fixed by moving the build context to the repo root (`docker build -f services/api/Dockerfile .`, both in `ci.yml`'s `docker-build` job and `deploy.yml`) and adding a root `.dockerignore`. Verified by deliberately running the old command first, watching it fail with exactly that error, then fixing and re-running — not assumed fixed because the Dockerfile changed.

## Addendum: the real production pass rate was much lower than the pre-deployment estimate — and the fix was not what was tried first

The pre-deployment estimate (~29% first-attempt pass rate, 7 samples) was flagged in this same ADR as small-sample noise, not a measurement — that caution turned out to be exactly right. In real production use, players started hitting "could not generate a valid case after 5 attempts" often enough to notice and report it. Checking `gcloud logging read` against the deployed service's own cost logs (see the "found by testing" note above about why that logging exists at all) showed 5 of the last 8 real requests fully exhausted all 5 attempts, and every exhausted request's token usage was consistent with never reaching the safety-check stage — i.e. 100% of observed failures were logic-consistency rejections, matching what the player reported ("it always fails at Checking logical solvability").

**Root cause, confirmed by reading real rejected candidates, not assumed:** manually generated and inspected several real candidates directly against `LiveCaseGenerator`. The pattern was consistent and genuine, not a false judge alarm — e.g. a candidate whose `solution.explanation` said "the vanilla smell matches Henrietta's favorite dessert," where no clue or suspect statement anywhere actually said Henrietta liked vanilla or baking; the connection existed only in the explanation, never in the case's own evidence. This is precisely the "hidden-only solution" the logic judge exists to catch, and it was catching real instances of it, correctly, most of the time. This also explains a secondary, separate finding: the judge's `reasons` output was identical, word-for-word, across completely unrelated case topics — because `evaluate_case_logic_v1.md`'s own prompt used those exact strings as its illustrative examples, and the judge was echoing them instead of describing the specific candidate.

**Two prompt fixes shipped, both real improvements, neither sufficient alone:**
- `evaluate_case_logic_v2.md` — asks for candidate-specific reasons and shows what that looks like instead of generic category labels. Confirmed fixed: every rejection reason logged in the investigation below is now unique and references actual candidate content.
- `generate_case_v2.md` — explicit clue-authoring guidance plus a worked good/bad example, aimed directly at the diagnosed failure pattern (put the identifying detail in the clue itself, never introduce it for the first time in the explanation).

**These did not measurably move the pass rate, tested rigorously, not assumed to have worked:** a 40-real-API-call investigation (not a token budget spent lightly, but the only way to actually know) tested the fix across two independent variables — prompt version (v1-style vague guidance vs. v2 with the worked example) and generator temperature (0.6 vs. the previous unset/~1.0 default) — specifically to isolate which lever mattered. Results: 0/10, 1/10, 0/10 across three 10-sample runs with the v2 prompt (two different temperatures) — no combination tested cleared roughly 10% first-attempt success, and the *specific* failure pattern (an unstated property "matched" to the culprit only in the explanation) persisted essentially unchanged in every rejected sample, worked example notwithstanding. gpt-4o-mini did not reliably generalize the pattern from either prose rules or a concrete example. This is a real, negative result, reported as one — not glossed over as "fixed" because the fix was reasonable and shipped.

**The pragmatic response, given trivial real cost:** rather than continuing to spend API calls chasing a prompt-engineering fix with no confirmed lever, `MAX_ATTEMPTS_PER_REQUEST` was raised from 5 to 10 and `DEFAULT_DAILY_ATTEMPT_CAP` from 300 to 2000. At a real per-attempt cost of ~$0.0006, even 2000 attempts/day is ~$1.20 worst case — trivial against the €10 budget alert regardless of exactly how low the true pass rate turns out to be. This does not fix content quality; it makes the feature tolerate the real quality ceiling of this model at this price point without the player hitting a dead end as often. A materially more capable generation model (e.g. gpt-4o instead of gpt-4o-mini) would likely fix this more thoroughly and remains a real option — not pursued in this pass because the cost/latency trade-off deserved its own deliberate decision, not a reflexive upgrade under time pressure.

**What's now true, stated plainly:** the real observed pass rate for this generator+prompt combination is closer to 5-10% per attempt than the originally estimated 29%. This ADR's earlier retry-count reasoning and the original `DEFAULT_DAILY_ATTEMPT_CAP=300` are both superseded by this addendum, not silently edited away — the numbers above (10 attempts, 2000/day cap) are the current, real ones.
