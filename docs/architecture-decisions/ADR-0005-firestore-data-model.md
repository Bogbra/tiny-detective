# ADR-0005: Firestore Data Model and Persistence

## Status

Accepted

## Context

Through Phase 6, all backend state (cases, players, hint requests) lived in process-local in-memory repositories behind the `CaseRepository`/`PlayerRepository`/`HintRequestRepository` Protocol interfaces defined in `app/application/ports.py` since Phase 3 ‚Äî deliberately, so the persistence layer could be swapped later without touching use cases or API routes. Phase 7 is that swap: wire real Firestore behind the same interfaces, plus the two collections the project spec's Firestore Data Model names that nothing was writing to yet (`case_attempts`, and full `hint_requests` records instead of a bare counter).

No real GCP project was available in this environment. The choice was between a real GCP Firestore project (needs an external account, a service-account key ‚Äî a materially more sensitive credential than the OpenAI key, since it can carry broad cloud permissions) and the local Firestore emulator. The user chose the emulator, run via Docker (`gcr.io/google.com/cloudsdktool/cloud-sdk:emulators`) ‚Äî the same client library and wire protocol as real Firestore, fully local, no cloud account needed.

## Decision

**Pure mapping, separate from I/O.** Each entity gets a `*_mapper.py` (`case_mapper.py`, `player_mapper.py`, `hint_request_mapper.py`, `attempt_mapper.py`) with plain `entity_to_document(entity) -> dict` / `document_to_entity(id, dict) -> Entity` functions ‚Äî zero Firestore imports, zero network calls. These alone satisfy Phase 7's "repository mapping tests" requirement, deterministically, in ~0.02s, with no emulator needed. The actual `Firestore*Repository` classes (`app/infrastructure/firestore/`) are thin: they call the client and hand data through the mapper. This mirrors the split ADR-0003 used for the AI pipeline (pure schema/rule stages vs. AI-calling stages) and ADR-0004 used for hint guardrails (pure `HintGuardrailPolicy` vs. the AI call) ‚Äî separate what can be tested without a network call from what can't, and don't make the former depend on the latter.

**Graceful degradation, same pattern as `OPENAI_API_KEY`.** `is_firestore_configured()` (`firestore_client.py`) gates whether `app/api/dependencies.py` wires Firestore-backed or in-memory repositories. Unset `FIRESTORE_EMULATOR_HOST`/`GOOGLE_CLOUD_PROJECT` means in-memory, not an error ‚Äî the app still runs and is still testable with zero external setup.

**Firestore-vs-tests needed a different guard shape than OpenAI's.** The OpenAI client is constructed lazily, inside a request handler, so a guard that raises when `"pytest" in sys.modules` (see below) fails safely ‚Äî the caller's existing `except Exception: return None` degrades to the deterministic fallback. Repositories here are constructed **eagerly at module-import time** in `dependencies.py` (so the singleton is ready before the first request). A raising guard at that point would crash test collection, not degrade ‚Äî so `is_firestore_configured()` itself returns `False` under pytest, steering `dependencies.py` to the in-memory repositories before any Firestore client is ever constructed.

**`daily_cases/{yyyy-mm-dd}`, not a single pointer.** The in-memory `CaseRepository`'s `set_daily`/`get_daily` tracked one case ID in a field. `FirestoreCaseRepository` implements the project spec's actual date-keyed collection ‚Äî more faithful to the spec, and it comes for free once real persistence exists (an in-memory equivalent would need its own dict-of-dates, not worth building for a repository that resets on every process restart anyway).

**AI-evaluation metadata (`evalScore`, `evalNotes`, `safetyFlags`, `ambiguityScore`, `promptVersion`, `modelName` on `cases/{caseId}`) is out of scope here.** The `DetectiveCase` domain entity (Phase 2) doesn't carry these fields ‚Äî they're Phase 5's concern, tracked in `tools/ai-content`'s local draft JSON records (`promptVersions: {generator, logic, safety}`, per ADR-0003's addendum), not synced into the backend's Firestore case documents. Bridging a draft into a real `cases/{caseId}` document with that metadata is `import_cases.py`'s job, still deferred ‚Äî there's a real persistent repository to import into now, but building the importer wasn't asked for this phase.

## Consequences

- Every use case (`SubmitSolution`, `RequestHint`, `ApproveCase`, `PublishDailyCase`, ...) works unmodified against either backend ‚Äî verified directly: `tests/integration/test_use_cases_with_firestore.py` runs the *same* use case classes against real Firestore-backed repositories, no use-case code changes needed, only the repository implementations are new.
- `HintRequestRepository.record()` and `AttemptRepository.record()` now take full domain entities (`HintRequest`, `Attempt`) instead of bare parameters ‚Äî a breaking change to the in-memory implementations and `RequestHint`/`SubmitSolution`'s internals, made once, before either had more than one caller. `count_for_case` on the in-memory hint repository is now derived by filtering stored records instead of a separate counter dict ‚Äî one source of truth instead of two data structures that could drift.
- The emulator persists data across test runs within one container's lifetime (unlike in-memory repositories, which reset every process) ‚Äî `tests/integration/conftest.py` clears the collections it touches before *and* after each test so runs don't see or leave stale data.
- No real GCP project was exercised ‚Äî only the emulator. Real-Firestore-specific behavior (IAM, quotas, network latency, security rules) is unverified; Phase 8 (deployment) is where that gap would need closing if this goes further than a portfolio demo.

## Addendum: a guard that "worked" was actually still broken ‚Äî found by testing the failure mode, not assuming it

Following this project's own pattern (Phase 5 and 6 both found real bugs by testing the thing that was supposed to already be safe, not by inspection): after wiring `is_firestore_configured()` to check `PYTEST_CURRENT_TEST` (mirroring the OpenAI guard), the fix was verified by deliberately stopping the emulator container and re-running the suite ‚Äî it hung for ~45 seconds instead of failing fast or falling back.

**Root cause**: `PYTEST_CURRENT_TEST` is only set by pytest while an individual test's setup/call phase is executing ‚Äî it is *not* set during pytest's collection phase, when test modules and their imports (including `app.api.dependencies`, transitively, via `app.main`) are first loaded. At that moment, the guard returned `True` (Firestore "configured"), so `dependencies.py`'s eager `_build_case_repository()` attempted a real connection to a stopped emulator, and the underlying gRPC client's connection/retry behavior against a closed port did not fail fast.

**Fix**: check `"pytest" in sys.modules` instead ‚Äî true from the moment the pytest process starts, before collection begins, not scoped to individual test windows. Applied consistently to all three guards in this project (this one, plus both OpenAI client guards in `services/api` and `tools/ai-content`) even though the OpenAI guards weren't actually broken by this specific gap (their real-client construction is lazy, inside a request handler, not at import time) ‚Äî kept them consistent anyway, since the weaker check (`PYTEST_CURRENT_TEST`) has the same latent gap if either ever became eager in the future.

Also fixed in the same pass: the integration tests' emulator-reachability probe originally used a real Firestore client call to check reachability ‚Äî also measured to hang against a stopped-but-not-yet-fully-torn-down container, for the same underlying gRPC-retry reason. Replaced with a raw `socket.create_connection(..., timeout=1.0)` TCP probe, which either succeeds or raises within one second, no gRPC involved at all for the check itself.

## Addendum: eager construction was the actual design flaw ‚Äî the sys.modules guard was the symptom fix

The `PYTEST_CURRENT_TEST` ‚Üí `sys.modules` fix above (and this ADR's own ¬ß"Firestore-vs-tests needed a different guard shape than OpenAI's") correctly diagnosed *why* the guard shape had to differ between OpenAI and Firestore: OpenAI's client is built lazily inside a request handler, Firestore's repositories were built eagerly at module-import time. Diagnosing that difference and then adding a stronger guard around the eager path treats the symptom ‚Äî it doesn't remove the reason a guard was needed there at all.

**The actual fix**: `app/api/dependencies.py`'s four repository providers (`get_case_repository`, `get_player_repository`, `get_hint_request_repository`, `get_attempt_repository`) and `get_hint_assistant` are now `@lru_cache`-decorated functions instead of eager module-level assignments. Nothing that talks to a network ‚Äî the Firestore client, the OpenAI client ‚Äî is constructed merely by importing `dependencies.py`; construction happens on first call, i.e. the first time a route actually resolves that `Depends()`. `functools.lru_cache` on a no-argument function gives the same "one instance per process, reused across requests" property the eager singletons had, without paying for it at import time.

This doesn't just fix the one bug the addendum above documents (the 45-second hang against a stopped emulator during test collection) ‚Äî it removes the entire class of import-time-network-call bugs, including ones that were never actually hit. `is_firestore_configured()`'s `"pytest" in sys.modules` guard is kept, not removed: it now protects a narrower, still-real case ‚Äî a test that calls the real (uncached-per-test) singleton without going through `app.dependency_overrides`, which would otherwise construct a real Firestore client lazily *during* a test run instead of at collection time. Verified: importing `app.api.dependencies` fresh (plain `python -c`, not pytest) with `FIRESTORE_EMULATOR_HOST` pointing at a stopped emulator completes in ~0.5s (dominated by normal import overhead, not a network attempt); calling `get_case_repository()` explicitly afterward is what triggers the real connection attempt, and only then.
