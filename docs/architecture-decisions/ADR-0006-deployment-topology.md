# ADR-0006: Deployment Topology, CI/CD, and Public-Demo Cost Control

## Status

Accepted

## Context

Phase 8 is real deployment: Firebase Hosting for `apps/game`, Cloud Run for `services/api`, real (non-emulator) Firestore, and GitHub Actions CI/CD — the last piece the project spec names, and the only one that had never been exercised end-to-end until now (Phases 1-7 verified everything against local tooling: the Firestore emulator, `flutter run -d chrome`, `uvicorn --reload`). No GCP or Firebase project existed before this phase; both were created live, on a private (non-Workspace) Google account with a freshly-created Billing Account, region `europe-west1`.

Before any of that: the backend exposes a public, unauthenticated, OpenAI-backed `/hint` endpoint. Once its URL is in the README, anyone can call it, and every call costs real tokens — the project spec's "Add rate limiting if endpoints are abused" stops being a hypothetical the moment the URL is public. This ADR treats that as a precondition of deploying at all, not a follow-up.

## Decision

**Three independent layers of cost/abuse control, not one.** A single safeguard is a single point of failure:
1. **Per-IP rate limiting on `/hint`** (`app/api/rate_limiting.py`, `slowapi`, 5/minute) — bounds how fast any one caller can spend tokens, independent of the existing domain-level `HintPolicy` (which bounds how much of *one case* a *player* can see, not how fast an *IP* can hit the endpoint at all). Keyed on `X-Forwarded-For`, not the raw socket peer: Cloud Run terminates the real client connection at its own proxy layer, so `request.client.host` inside the container reflects Cloud Run's internal hop, not the caller — every request would look like the same source, which would either never trip the limit or (worse) rate-limit every real user as one. Verified live against the deployed service, not just in tests: 5 requests from 5 different players succeeded, the 6th got `429` — confirming the `X-Forwarded-For` handling actually works through Cloud Run's real proxy, which local/emulator testing could never confirm.
2. **`--max-instances=3` on the Cloud Run service** — bounds the blast radius of a traffic spike regardless of what causes it. `--min-instances=0` is passed explicitly alongside it, not left at the implicit default, even though Cloud Run's own API default already is 0 — a demo should provably scale to zero when idle (cold starts don't matter here; a permanently-running instance with no traffic does). One consequence worth naming honestly: Cloud Run's Knative-style annotations only get written when a value differs from the platform default, so `gcloud run services describe` shows `autoscaling.knative.dev/maxScale: '3'` but no `minScale` annotation at all — its *absence* is what "0" looks like, not a sign the flag didn't apply. Confirmed the flag was honored by checking the deployed service directly, not by assuming the deploy command succeeding meant the setting took effect.
3. **A €10 GCP budget alert** (50/90/100% thresholds, email to the billing account admin) — the outlier case that per-IP limiting and instance caps don't fully rule out (a botnet spreading requests across many IPs, a pricing change, a bug in the limiter itself). Free tier covers normal demo traffic completely; this is insurance against the tail, not the expected case.

**Workload Identity Federation for Cloud Run, a stored service-account key for Firebase Hosting — an honest trade-off, not a single principle.** `github-actions-deployer` authenticates to GCP via WIF (`google-github-actions/auth@v2`): no long-lived credential exists anywhere, GitHub's own short-lived OIDC token is exchanged for GCP access at run time, scoped by an attribute condition to `Bogbra/tiny-detective` specifically. Firebase Hosting deploys, by contrast, use a JSON key for a dedicated `firebase-hosting-deployer` service account (`roles/firebasehosting.admin` only), stored as the `FIREBASE_SERVICE_ACCOUNT` GitHub secret. This is deliberately not glossed over as "WIF because no long-lived key" as a blanket rule — it would contradict itself, since a key *is* stored for Hosting. The honest framing: Cloud Run access (WIF-native, well-documented via `google-github-actions/auth`) and Firebase Hosting access (conventionally done via a scoped service-account key, e.g. `firebase init hosting:github`'s own generated flow) are two different targets with two different idiomatic paths in their respective ecosystems, and only one of those paths currently avoids stored key material. The key that does exist is scoped to nothing but Hosting deploys for this one repo.

**Two Cloud Run-side service accounts, not one** — `github-actions-deployer` (deploy-time: `roles/run.admin`, `roles/artifactregistry.writer`, `roles/iam.serviceAccountUser`, `roles/firebaserules.admin`, `roles/datastore.indexAdmin`) is a different identity from `tiny-detective-api-runtime` (what the container actually runs as: `roles/datastore.user`, plus `roles/secretmanager.secretAccessor` bound to exactly the two secrets it needs, not a project-wide grant — the first attempt granted the role at the project level and was corrected to per-secret bindings once noticed). Separating "who can deploy" from "what the running service can touch" means a compromised deploy pipeline and a compromised running container have different blast radii.

**Runtime secrets (`ADMIN_API_TOKEN`, `OPENAI_API_KEY`) live in Secret Manager, referenced via `--set-secrets`, never passed as plain `--set-env-vars`.** The production `ADMIN_API_TOKEN` is a freshly-generated value distinct from the local-dev `devsecret`, created and stored without ever being printed to any command output or CI log.

**`GOOGLE_CLOUD_PROJECT` is set explicitly as a Cloud Run env var.** Unlike some other GCP compute products, Cloud Run does not auto-inject this — without it, `is_firestore_configured()` (see ADR-0005) would silently fall back to in-memory storage in production, which would still *work* (nothing crashes) but would silently defeat the entire point of Phase 7. This is exactly the kind of quiet, plausible-looking failure the project has run into before with eager construction and pytest guards — caught here by checking, not by assuming the deploy would fail loudly if it were wrong.

**One `deploy.yml`, two jobs, not three chained workflow files.** The project spec's monorepo listing names `ci.yml`, `deploy-web.yml`, `deploy-api.yml` as separate files. A `workflow_run`-chained pipeline (`ci.yml` → `deploy-api.yml` → `deploy-web.yml`) was the first design, then simplified after review: `workflow_run` always reads the *triggering* workflow's definition from the default branch (not the commit being built), multi-level trigger chains are hard to follow, and a broken link between three separate workflow runs is easy to miss in the Actions UI. A single `deploy.yml`, triggered once by CI's `workflow_run`, with `deploy-web` declaring `needs: deploy-api` for ordering, gives the identical guarantee — the frontend never builds against a backend that isn't live yet — in one visible run, with `deploy-web` reading `deploy-api`'s Cloud Run URL directly via job outputs instead of re-deriving it with a separate `gcloud run services describe` call. This is a deliberate, documented deviation from the spec's literal file listing, not an oversight — the same "simplest solution that meets the requirement" principle the project has applied everywhere else (e.g. Phase 1's deferred per-feature directory trees, ADR-0004's backend-local prompts).

**`contract-checks` is a deliberately redundant CI job.** It re-runs `tests/contract/` on its own, a strict subset of what `backend-checks` already covers running the full suite. The project spec names it explicitly as a required job, and a PR showing a separately-labeled "contract-checks failed" states exactly what broke (the public API's contract) rather than "backend-checks failed," which could mean anything from a domain unit test to this. The CI-time cost is small (both jobs run in well under 30 seconds); the signal clarity is judged worth it.

**`firestore.rules` denies all direct client access.** The backend's `google-cloud-firestore` server SDK, authenticated as `tiny-detective-api-runtime` via project-level IAM, bypasses Security Rules entirely — these rules have zero effect on how the deployed app actually behaves. They exist to close the gap `docs/scalability.md` already documented as unverified: nothing previously stopped a client SDK from talking to Firestore directly if that architectural rule were ever violated by mistake. `firestore.indexes.json` is empty, matching the single-field-only query shapes already documented there.

## Consequences

- The live demo URL (`https://tiny-detective-ai.web.app`) is now real and was verified end-to-end: a full browser playthrough (load the daily case → request a hint from the real OpenAI-backed assistant, grounded and non-revealing → submit the correct suspect → see the scored result) succeeds against the actually-deployed Cloud Run backend and real Firestore, not the emulator.
- The composite-index and Security Rules gaps `docs/scalability.md` flagged as "locally verified, index/rules behavior unverified" are now partially closed: Security Rules are deployed (even though bypassed by the server SDK) and today's query shapes still need no composite index against real Firestore — confirmed by the deploy actually succeeding, not just by the rules file existing.
- A real Billing Account now exists on the project owner's Google account, with a budget alert as the safety net, not a hard cap — GCP budget alerts notify, they do not automatically stop spend. If the free tier were ever meaningfully exceeded, someone still has to notice the email and act; this is accepted as adequate for a portfolio demo's traffic level, not a claim that spend is capped at €10.
- `firebase-hosting-deployer`'s JSON key is the one piece of long-lived credential material in this whole pipeline. Rotating or revoking it is a manual `gcloud iam service-accounts keys` operation plus a `gh secret set`, not automatic — acceptable for a single-repo demo, would need real key-rotation tooling before this pattern scaled to more services.
