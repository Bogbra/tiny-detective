# Final Architecture And Quality Review

All 8 phases from the project spec's "Test Planning By Phase" are complete: Project Scaffolding, Core Domain Rules, Backend API, Flutter Game UI, AI Case Generation, AI Hint Assistant, Persistence and Firestore, CI/CD and Deployment. This review is the closing check required before the project counts as portfolio-ready — see `docs/phase-reviews.md` for the full phase-by-phase history this summarizes.

## Safety

**Secrets.** No API key, admin token, or credential has ever been committed — `.gitignore` blocked `.env` from day one, and the two production secrets created in Phase 8 (`ADMIN_API_TOKEN`, `OPENAI_API_KEY`) were piped directly into `gcloud secrets create --data-file=-` without ever being echoed to any command output or CI log. They live in Secret Manager, injected into Cloud Run via `--set-secrets`, never as plain environment variables. AI provider keys are backend/tooling-only — the Flutter app has never called an AI provider or Firestore directly in any phase; both go through `services/api`.

**Solution and private-suspect-field leakage.** Enforced at the type level, not by a runtime filter that could be forgotten: `PublicDetectiveCase`/`PublicSuspect`/`PublicClue` (Phase 2) simply don't have `solution`, `isCulprit`, or `privateReasoning` fields to leak. Verified four separate times across the project's life: Phase 2's pure unit tests, Phase 3's contract tests against in-memory repositories, Phase 7's integration test proving the same guarantee holds after a real Firestore round-trip, and Phase 8's live `curl` against the actually-deployed public URL.

**Admin endpoints.** Disabled by default (no `ADMIN_API_TOKEN` set → every request `401`, regardless of what header is sent) rather than "protected by a guessable default." Comparison uses `secrets.compare_digest` to avoid a timing side-channel. Verified live in Phase 8 against the real deployed backend, not just local tests.

**The AI hint assistant.** Structurally cannot invent facts or reveal the culprit by name/id, because it only ever receives `case.public_view()` — the culprit's identity is never in its context to leak. A separate deterministic guardrail (`HintGuardrailPolicy`) additionally screens the generated text for suspect names and roles, with the role check scoped to whether the role word is grounded in the specific clue being referenced (not a blanket ban), closing a real gap a names-only version left open (Phase 6's "First fix (wrong)" — documented rather than hidden). A deterministic fallback hint fires on any AI failure or guardrail violation.

**Public-demo cost/abuse control (Phase 8, the newest safety surface).** The `/hint` endpoint costs real OpenAI tokens per call and has no authentication — addressed with three independent, layered controls rather than one: per-IP rate limiting (5/minute, keyed on `X-Forwarded-For` since Cloud Run's proxy layer would otherwise make every caller look identical), `--max-instances=3` on Cloud Run, and a €10 budget alert. All three verified against the live deployment, not assumed from the deploy command succeeding.

**Known, accepted gaps** (documented, not hidden): admin auth is a single shared-secret header, not real authentication — adequate for an MVP with no real users, would need replacing before real admin access is exposed. `firestore.rules` denies all direct client access but that enforcement path is structurally never exercised, since the backend's server SDK bypasses Security Rules via IAM — the rules exist as a backstop against a violation of the "Flutter never talks to Firestore directly" architecture rule, not as an active control today.

## Efficiency

The project consistently chose the simplest solution that met the actual requirement, and named the alternative it didn't take when that mattered: in-memory repositories until Phase 7 actually needed persistence (not built ahead of need); no per-feature Flutter directory trees until a feature existed to put in them (Phase 1); only two of five AI evaluation stages call a model at all, the rest are deterministic Python (Phase 5); Workload Identity Federation for Cloud Run but a conventional stored key for Firebase Hosting, because that's genuinely how each ecosystem's GitHub Actions integration idiomatically works, not because one universal rule was applied inconsistently (Phase 8, ADR-0006); a single `deploy.yml` with two ordered jobs instead of the spec's literal three-workflow-file listing, simplified specifically because the more complex version added fragility without a corresponding benefit at this project's size.

Where the project deliberately paid a complexity cost, it was for a stated reason that later paid off: the port/adapter interfaces (`app/application/ports.py`) since Phase 3 meant Phase 7's real Firestore swap needed zero use-case or route changes — the same use case classes run unmodified against Firestore-backed repositories, proven directly in `test_use_cases_with_firestore.py`. Clean layering throughout both the Flutter app and the backend costs more files than a flatter structure would, chosen explicitly because demonstrating that discipline is this project's actual goal (ADR-0002).

Two accepted, named inefficiencies: `contract-checks` in CI re-runs a strict subset of what `backend-checks` already covers, in exchange for a precisely-labeled PR check (Phase 8); `tools/ai-content` and `services/api` each depend on the OpenAI SDK independently rather than sharing it, because they're genuinely different runtime contexts (offline batch tooling vs. a live in-game request) (Phase 6, ADR-0004).

## Regression Risks

The most likely regressions and what actually guards against each:

- **A future change reintroduces solution/private-field leakage.** Guarded by the type-level split (impossible to leak a field the public type doesn't have) plus four independent test layers (unit, contract, Firestore-integration, live production) that would all need to be simultaneously wrong.
- **A future guardrail "fix" reopens the role-identification gap Phase 6 already found once.** Guarded by `tests/fixtures/hint_guardrail_cases.json` and the ADR-0004 addendum explicitly naming the wrong-first-attempt and why it was wrong — the next person changing this code has the failure mode spelled out, not just the current passing state.
- **A dependency-wiring change reintroduces eager network-client construction at import time**, the root cause behind Phase 7's 45-second pytest hang. Guarded by the `@lru_cache`-based lazy construction pattern in `app/api/dependencies.py` (ADR-0005's second addendum) plus the `sys.modules` pytest guard as defense-in-depth.
- **Rate limiting silently breaks under concurrent test runs or gets bypassed by a future route that forgets the decorator.** Partially guarded (the reset-fixture regression was caught red-then-green in Phase 8), but there is no test asserting *every* cost-bearing route has rate limiting — today that's just `/hint`, and it would need conscious extension if another AI-calling endpoint were added.
- **A future deploy accidentally omits `GOOGLE_CLOUD_PROJECT` or a `--set-secrets` binding.** Not guarded by an automated test today — this is deploy-configuration state living in `deploy.yml`, not application code; the mitigation is the `/health` smoke test failing the deploy job, plus the live Firestore-document check performed manually in Phase 8. A stronger guard (e.g. asserting the deployed service's env vars via `gcloud run services describe` in CI) doesn't exist yet.

## Remaining Tests

- **Unit / domain**: complete for all Phase 2 policies (scoring, hint limit, publish eligibility, case consistency, solution validation) and Phase 8's rate limiter.
- **Contract**: complete for all public and admin endpoints, including the two Phase 8 additions (rate-limit 429, and re-verifying solution/private-field non-leakage still holds).
- **Integration**: complete for all four Firestore repositories against the real emulator; not yet extended to a real GCP project's IAM/quota/latency behavior beyond what Phase 8's actual live deploy incidentally exercised.
- **AI eval / guardrail**: complete for the golden/invalid case sets (8 golden cases across two difficulty tiers) and the hint guardrail fixture; the automated eval suite still uses fake evaluators for speed — `evaluate_cases.py` exists specifically as the separate, cost-aware script for spending real API calls to check judge quality, and was last run manually, not on a schedule.
- **UI / widget**: complete for the built screens (case, suspects, clues, hints, result, loading/error states); no true end-to-end Flutter integration-test suite exists — Phase 4 and Phase 8 both verified real behavior via Playwright browser automation against a running/deployed instance instead, which is real coverage but not part of `flutter test`.
- **Manual / exploratory, still open**: no load test validates `--max-instances=3` under genuine concurrent traffic (only single-caller rate-limit behavior was checked); `firestore.rules`' actual enforcement has never been triggered by a real forbidden client request, since nothing in this architecture is supposed to attempt one; daily-case publishing in production is still a manual admin call with no scheduled automation.

## Portfolio Readiness

The project demonstrates what it set out to: a playable Flutter game (`apps/game`, live at the Hosting URL) backed by a fullstack service (`services/api`) with clean domain/application/infrastructure layering enforced and verified (zero framework imports in the domain layer, checked by grep, every phase). AI is visibly part of the product architecture in the three places the spec named — case generation, multi-stage evaluation, and a grounded, guardrailed hint assistant — not bolted on as a demo gimmick, with the evaluation pipeline's own quality measured (precision/recall/specificity against a golden set) rather than assumed. Guardrails are designed, tested, and were caught failing at least twice during development (the role-identification gap, the eager-Firestore-construction hang) with both failures documented rather than smoothed over in hindsight — arguably the strongest portfolio signal in the repo, since it shows the review process actually catching real defects, not just producing paperwork. Tools, services, and app code stay in clearly separate directories with explicit contracts between them. CI/CD is real: five gating checks plus a two-job deploy pipeline that has actually run successfully against real cloud infrastructure, not just written and left unexercised. Deployment is live and was verified end-to-end via an actual browser playthrough against production, not inferred from a green CI run. Scalability and trade-off notes are honest about what's actually been verified versus what's been reasoned through but not exercised (`docs/scalability.md`'s explicit "locally verified, X unverified" framing, updated again in Phase 8 rather than left stale).
