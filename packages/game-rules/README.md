# Game Rules

Shared, framework-independent game-rule logic that both `apps/game` and `services/api` may need to agree on (e.g. difficulty definitions, scoring constants), kept as a single source of truth instead of duplicated.

**Still intentionally empty — the anticipated trigger never actually happened.** All scoring, difficulty, hint-limit, and solution-validation logic lives once, in `services/api/app/domain/policies/`, computed server-side and returned to `apps/game` as data (a score, a streak, a result) — the Flutter side displays what the backend computed rather than independently re-implementing any of that logic, so the cross-client duplication this package exists to prevent never actually arose. This was a deliberate bet made early (see the original note this replaces: "only if actual duplication between frontend and backend shows up — not created preemptively") that turned out correct once the project was built out. Revisit if a future client (a native mobile app, say) ever needs to validate a solution or compute a score locally, offline, without a round-trip to the backend — that's the concrete scenario that would justify populating this.
